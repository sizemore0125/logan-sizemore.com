Configuring Git

Set User Details:
It is important for you to specify your name and your email so that the changes that you make can be attributed to you.

git config --global user.name "Your Name"
git config --global user.email "your_email@example.com"

Default Editor:
By default, Git uses Vim as the default text editor, but you can change this. For example, if you want to change your default editor to VSCode, you can run this line. 

git config --global core.editor "code"

Handling Line Endings:
Windows: We need to ensure that line endings are converted to CRLF when checked out and normalized to LF when committed. This prevents compatibility issues between systems.

git config --global core.autocrlf true

Mac/Linux: We need to normalize line endings to LF during commits. This is the standard for Unix-based systems. Again, this helps prevents compatibility issues between systems.

git config --global core.autocrlf input

Vim: A Quick Aside

Starting and Exiting Vim:

Open a file in Vim:

vim filename

Exit without saving:

Save without exiting:

Modes in Vim

Vim has two main modes:

  1. Normal Mode:
    • For navigation and commands.
    • Press Esc to re-enter Normal mode.
  2. Insert Mode:
    • For typing and editing text.
    • Enter Insert mode by pressing:
      • i (insert before the cursor)
      • SHIFT + a (append at the end of the line)
      • o (open a new line below the cursor)

Navigation (In Normal Mode)

Move cursor:

  • h: left
  • l: right
  • j: down
  • k: up
    Jump words:
  • w: next word
  • b:previous word
    Move to line start or end:
  • 0: start of the line (If you type a number, it will go to that character)
  • $: end of the line
    Go to a specific line:
:linenumber

Initializing a Git Repository

Create a directory for your project on Mac/Linux (or GitBash on Windows) using:

mkdir <directory_name>
cd <directory_name>

You can then initialize a new Git repository:

git init

When you do this, it will create a creates a hidden .git directory that will contain your project history. If you want to view this directory, you can list hidden files with:

ls -la

You will see something like:

drwxr-xr-x 1 sizem 197609   0 Jan  1 12:01 ./
drwxr-xr-x 1 sizem 197609   0 Jan  1 12:04 ../
drwxr-xr-x 1 sizem 197609   0 Jan  1 12:04 .git/

Notice the .git directory. This was created when you ran git init.

DO NOT modify the .git directory.

Git Workflow

The Git Workflow was covered in the slides, and you will get practice with this in lab.
Workflow Steps::

  1. Modify files in the working directory.
  2. Stage changes the staging area changes using the git add command.
  3. Commit changes in the staging are to save a snapshot in the repository.

Example Workflow:

Step 1: Modify files in the working directory.

Create some file in your working directory.

touch Hello.java

Then, I will use Vim to write this code.

public class Hello {
	public static void main(String[] args) {
		System.out.println("Hello, World!");
	}
}

Then, I will compile and run this code to confirm that it works.
In the terminal, to compile Hello.java:

javac Hello.java

Then, I can run my program:

java Hello

And this will display:

Hello, World!

This is all looks good, so I want to save this change to my project.

Step 2: Stage changes the staging area.

We can view the status of changes using git status:

git status

Which will show us something like this:

On branch master

No commits yet

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        Hello.class
        Hello.java

nothing added to commit but untracked files present (use "git add" to track)

This tells us that we have two untracked files: Hello.java, the java code we just wrote, and Hello.class the java byte-code we got when we compiled Hello.java.

Note on Git Status:
You can also use git status -s and this will show you a shortened version of git status. For example:

?? Hello.class
?? Hello.java

This shows that I have two untracked files: Hello.class and Hello.java.

I don't really want to keep Hello.class, so I am only going to add the code I wrote to the staging area:

git add Hello.java

Now, when I view the staged changes with git status, I see this:

On branch master

No commits yet

Changes to be committed:
  (use "git rm --cached <file>..." to unstage)
        new file:   Hello.java

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        Hello.class

This shows that Hello.java is ready to be committed to my repository, and Hello.class is still unstaged, and will not be committed (unless I stage it).

Note on git add:
We can add one file, like what is shown above. Or, we can add multiple files:

git add File1.java File2.java

Or, we can add multiple files with patterns:

git add *.java

This will add all .java files.

Or, we can add all files in our directory:

git add .

Step 3: Commit changes

Let's now commit the file in our staging area:

git commit

When I run this, it will bring me to my default text editor (Vim in my case), and I am prompted to write some message to describe the changes I made to my project.

Alternatively, we can provide a message directly when we commit:

git commit -m "Added a Hello.java, which displays 'Hello, World'."

When we commit, we see something like:

[master (root-commit) a4312f4] Added a Hello.java, which displays 'Hello, World'.
 1 file changed, 5 insertions(+)
 create mode 100644 Hello.java

Now, when i check the status of my working directory using git status:

On branch master
Untracked files:
  (use "git add <file>..." to include in what will be committed)
        Hello.class

nothing added to commit but untracked files present (use "git add" to track)

We now no longer see the message No commits yet, meaning that our commit was successful, and we no longer see Hello.java because it is now saved in its current form in our git repo.

Note on Commit Messages:
It is best practice (really important) that you create meaningful commit messages. Please do not simply write "committing" or "asdfadsf" or anything like that. Instead, write a message that describes what you did. For example, "Fixed a bug where the user could not enter in floating point numbers."

Note on Commit Best Practices:
It is important that your commits are meaningful. This means that they are neither too big or too small, and that they represent logical changes to your project. For example, a commit may be a single bug fix, or the addition of a small feature, but a single commit should NOT contain both a bug fix and the addition of a small feature.

Viewing Commit History

Let's say that we want to change our program, Hello.java. 

public class Hello {
	public static void main(String[] args) {
		System.out.println("Hello, CSCI 145!");
		System.out.println("We are learning Git!");
	}
}

Now, when I check the status of my working directory using git status:

On branch master
Changes not staged for commit:
  (use "git add <file>..." to update what will be committed)
  (use "git restore <file>..." to discard changes in working directory)
        modified:   Hello.java

Untracked files:
  (use "git add <file>..." to include in what will be committed)
        Hello.class

no changes added to commit (use "git add" and/or "git commit -a")

I can see that my Hello.java file has been modified. So, I am going to add that file to the staging area, and then I am going to commit:

git add Hello.java
git commit -m "Changed Hello.java to greet CSCI 145 and convey that we are learning Git."

Now, let's say that I want to view my commit history. To do this we can use the git log command:

git log

This will show us:

commit b661a2bcb48a5ddac2d8c417dde8bdc5bb69d737 (HEAD -> master)
Author: Logan Sizemore <sizemore0125@gmail.com>
Date:   Wed Jan 1 14:22:00 2025 -0800

    Changed Hello.java to greet CSCI 145 and convey that we are learning Git.

commit a4312f455a46e1ee90be4da6c0b54a3dcca164f2
Author: Logan Sizemore <sizemore0125@gmail.com>
Date:   Wed Jan 1 13:51:24 2025 -0800

    Added a Hello.java, which displays 'Hello, World'.

Here we can see the commit history for our repository in reverse chronological order. Each commit includes:

  1. Commit Hash: A unique identifier for the commit.
  2. Author: The name and email address of the person who made the commit.
  3. Date and Time: The timestamp of when the commit was made.
  4. Commit Message: The commit message we made earlier.

Maybe this is too much information for us, and we just want a short description of each commit. In this case, we can use:

git log --oneline

Which shows us this:

b661a2b (HEAD -> master) Changed Hello.java to greet CSCI 145 and convey that we are learning Git.
a4312f4 Added a Hello.java, which displays 'Hello, World'.

This gives us a truncated commit hash and the one line commit message.

We can also view our git history in chronological order using the --reverse flag:

git log --oneline --reverse

Which shows:

a4312f4 Added a Hello.java, which displays 'Hello, World'.
b661a2b (HEAD -> master) Changed Hello.java to greet CSCI 145 and convey that we are learning Git.

Ignoring Files

When we compile our java code, we end up with one or more .class files, which we do not want to commit (In fact, I will mark you down on your assignments if you do).

So, we could simply avoid committing those files, or we could create a .gitignore file, which we can use to specify certain files we do not want to track.

First, let's make a .gitignore file:

touch .gitignore

And now, in that file I will list everything I do not want to track. In my case, I don't want to track Hello.class.

Hello.class

However, this does not help me if I ever have another .class file. So, instead I can write this:

*.class

Now, I won't track any files that end with .class.

Personally, whenever I start a project, I find a .gitignore template that works for me: https://github.com/github/gitignore

For our purposes, this .gitignore will be more than enough:

# Compiled class file
*.class

# Log file
*.log

# BlueJ files
*.ctxt

# Mobile Tools for Java (J2ME)
.mtj.tmp/

# Package Files #
*.jar
*.war
*.nar
*.ear
*.zip
*.tar.gz
*.rar

# virtual machine crash logs, see http://www.java.com/en/download/help/error_hotspot.xml
hs_err_pid*
replay_pid*

Remember to add and commit your .gitignore file after you make it.

Now, when we check the status of our working directory we no longer see our .class files:

On branch master
nothing to commit, working tree clean

Viewing Past Commits

As we make changes to our project, we may want to examine earlier versions of a file.

To do this, we can use the git show command. For example, if we want to see what Hello.java looked like in the first commit, we need to first identify the relevant commit hash using git log:

git log --oneline

This might produce output such as:

9eae365 (HEAD -> master) Created a .gitignore
b661a2b Changed Hello.java to greet CSCI 145 and convey that we are learning Git.
a4312f4 Added a Hello.java, which displays 'Hello, World'.

Using the hash a4312f4 from the first commit, we can view the file as it appeared in that commit:

git show a4312f4 -- Hello.java

This displays:

commit a4312f455a46e1ee90be4da6c0b54a3dcca164f2
Author: Logan Sizemore <sizemore0125@gmail.com>
Date:   Wed Jan 1 13:51:24 2025 -0800

    Added a Hello.java, which displays 'Hello, World'.

diff --git a/Hello.java b/Hello.java
new file mode 100644
index 0000000..b77c397
--- /dev/null
+++ b/Hello.java
@@ -0,0 +1,5 @@
+public class Hello {
+       public static void main(String[] args) {
+               System.out.println("Hello, World!");
+       }
+}

This output provides the commit details along with the file's contents and highlights any lines that were added or removed in that commit.

Viewing Differences Between Commits

In addition to viewing the history of a file, we can also use Git to compare changes between commits.

For instance, if we want to see how Hello.java has changed between the first commit (a4312f4) and the most recent commit, we can use the git diff command:

git diff a4312f4 -- Hello.java

This produces:

diff --git a/Hello.java b/Hello.java
index b77c397..e4256b3 100644
--- a/Hello.java
+++ b/Hello.java
@@ -1,5 +1,6 @@
 public class Hello {
        public static void main(String[] args) {
-               System.out.println("Hello, World!");
+               System.out.println("Hello, CSCI 145!");
+               System.out.println("I am learning Git!");
        }
 }

In this output:

  • Lines prefixed with - indicate content removed from the earlier commit.
  • Lines prefixed with + indicate content added in the newer commit.

This allows us to review all changes made to Hello.java between the specified commits.

Reverting Changes

Let's say that I delete Hello.java and commit that change:

rm Hello.java
git add Hello.java
git commit -m "Removed Hello.java"

However, perhaps this was a mistake, and we want to get Hello.java back.

In this case, we should use the git restore to restore a file to a previous version. To restore Hello.java we do this:

git restore --source=9eae365 Hello.java

Note: 9eae365 is the commit hash for the commit that we want to grab Hello.java from.

Now, we should see that our file is restored. However, it is currently untracked. So, we should add and commit this file.

git add Hello.java
git commit -m "Restored Hello.java."

Branches

Imagine you are working on a homework assignment and are satisfied with your code so far. It is stable and functions as expected. Now, you want to add a new feature but are concerned about disrupting your existing, well-functioning code. What should you do?

This is where branches become useful. Branches let you create a separate copy of your code where you can experiment and make commits without affecting the original code. Once you are satisfied with the changes, you can merge them back into the Main branch.

Note: By default, Git uses "Master" as the branch name, while GitHub uses "Main." The repositories created for this class will use "Main" as the default branch. If you hear me refer to "Main" or "Master," I am always referring to the default branch.

Creating Branches

To add a new feature, you can create a new branch named feature:

git branch feature

You can view all your branches with:

git branch

This might display:

  feature
* master

The * indicates the branch you are currently on.

To switch to the feature branch, use the checkout command:

git checkout feature

You will see:

Switched to branch 'feature'

You can now make changes to your code in the feature branch without affecting the master branch.

Merging Branches

Once the feature is complete and you are confident it works as expected, you can merge it back into the master branch.

First, switch back to the master branch:

git checkout master

Then merge the feature branch into the master branch:

git merge feature

The merge summary will appear, for example:

Updating 23e276c..81455fa
Fast-forward
 Hello.java | 6 ++++++
 big.txt    | 1 +
 2 files changed, 7 insertions(+)
 create mode 100644 Hello.java
 create mode 100644 big.txt

This process allows you to safely develop and test new features while maintaining the stability of your main branch.

Remote Repositories

In this class, you will submit assignments to a remote repository on GitHub that I will create for you. After you provide your GitHub username, I will invite you to the repository. On the assignment due date, an automated script will check your repository, and the TAs will grade your work.

Note: Git and GitHub are distinct. Git is a version control system, while GitHub is a platform for hosting Git repositories.

For example, a remote repository for this class might have the following URL:

https://github.com/sizemore-teaching/202510_csci145_sizemol

Here, sizemol will be replaced with your university username.

Cloning a Repository:

The repository exists on GitHub's servers, but you need a local copy to begin working. To achieve this, you will clone the repository:

git clone https://github.com/sizemore-teaching/202510_csci145_sizemol.git

You will see a message similar to:

Cloning into '202510_csci145_sizemol'...

Note: When cloning a private repository, you will be prompted to enter your GitHub username and a GitHub Personal Access Token. Detailed instructions can be found here.

Pushing and Pulling Changes:

If you work on an assignment at school but want to continue at home, you must first push your local changes to the remote repository on GitHub.

Assuming you have committed your changes, use the following command:

git push

This updates the remote repository on GitHub with your changes.

When you are at home, navigate to your local copy of the repository and synchronize it with the remote repository by running:

git pull

This ensures that your local repository at home matches the remote repository on GitHub.

Important: The TAs will grade assignments based on the content in your remote repository on GitHub. Always push your changes once you complete your assignment.
